# 文档组织规范

按角色编写
- 每篇文档只服务于一个角色：使用配置者（User）、技术决策者（Decision）、开发者（Dev）。严禁在同一文档混合多种视角。
- 在同一角色下，按“单一核心概念”组织（如“并行与分片”）；涉及其他概念请以链接引用，避免掺杂。

篇幅与写法
- 单文件不超过 300 行（含代码/表格）。超过请拆分为子主题，并提供该主题的简短索引页。
- 推荐结构：问题/目标 → 核心概念 → 步骤/决策/实现 → 最小示例 → 相关链接 → FAQ（可选、≤5 条）。
- 定义/Schema 的“单一事实源”放在 `docs/reference/`；其他文档只链接，不复制粘贴。

导航与结构
- 三个角色的索引目录：`docs/user/`、`docs/decision/`、`docs/dev/`。

命名与元信息
- 文件名使用 snake_case、简短易懂（如 `sinks_parallel_and_sharding.md`）。
- 文首用注释标注“角色/最近验证日期”。
- 正文不出现对外“v2/v3”等版本标签（代码路径中的真实目录名除外）。

链接规范
- 使用工作区相对路径，并用反引号包裹（如 `docs/user/config/sinks_design_and_config.md`）。
- 优先链接到“单一事实源”，不要重复定义。
- 必要时附代码路径与行号（如 `src/services/sink/act_sink.rs:214`）。

作者自检
- 篇幅 ≤ 300 行。
